# HBF 每层时间预算与预取停顿明细

## 1. 目的

本文档专门回答下面这个问题：

- 在“不卸载参数”的情况下，每一层完整执行到底需要多少时间
- 这个时间能为“下一层的预测 + HBF 传输”提供多少时间预算
- 在不同 HBF 配置下，每层的 `T_predict + T_transfer` 是多少，最终会产生多大的 stall

## 2. 统计口径

### 2.1 不卸载时每层总时长

这里的“每层总时长”不是只看某一个算子，而是看一个完整的 Transformer hidden layer 在当前 profile 下的总执行时长。

对 Llama-3.1-8B 的单层 hidden block，统计范围为：

- `input_layernorm`
- `q_proj`
- `k_proj`
- `v_proj`
- `rope`
- `attn`
- `o_proj`
- `post_layernorm`
- `gate_proj`
- `up_proj`
- `act_fn`
- `down_proj`

在当前测试配置中：

- `tp = 1`
- 单节点单实例
- 没有额外的张量并行通信
- 因此“不卸载”时每层总时长可以直接视为这一整层 hidden block 的总执行时间

### 2.2 预取时间与 stall

对 HBF 配置，每层预取总时长按：

`T_prefetch = T_predict + T_transfer`

其中：

- `T_predict = predict_base_ns + predict_attn_ns + predict_ffn_ns`
- `T_transfer = T_attn_transfer + T_ffn_transfer`

每层 stall 按：

`stall = max(0, T_prefetch - T_layer_total)`

这里：

- `T_layer_total` 就是不卸载时该层完整执行的总时长
- 也就是“下一层可用于预测+传输的时间预算”

## 3. 32 层不卸载时间预算

从当前 trace 中抽取的 32 个 hidden block 总时长完全一致：

- 每层总时长：`771319 ns`

这意味着：

- 对于第 `0 ~ 30` 层，可以把 `771319 ns` 看作“用于下一层预测+传输的时间预算”
- 第 `31` 层没有下一层 hidden block，因此不再需要为下一层做 prefetch

## 7. 结论

### 7.1 不卸载时的层预算

- 在当前 profile 下，32 个 hidden layer 的完整执行总时长一致，都是 `771319 ns`
- 这个值可以直接作为“下一层预测 + 传输”的时间预算

### 7.2 HBF 配置下的层级结论

- `HBF-1.0 / 0.75 / 0.50 / 0.25` 在当前层预算下都可以把预取完全隐藏住，因此逐层 stall 为 `0`
- `HBF-0.50-P2ms` 和 `HBF-0.50-P5ms` 的 `predict` 时间过大，超过了层预算，导致逐层 stall 分别为：
  - `1568883 ns`
  - `4568883 ns`

### 7.3 你真正关心的“可用时间”

如果你要判断某个 HBF 配置能不能把预取隐藏在当前层执行里，可以直接比较：

- `不卸载层总时长 = 771319 ns`
- `该配置的每层 T_prefetch`

判断规则是：

- 如果 `T_prefetch <= 771319 ns`，则该层不会因为预取产生 stall
- 如果 `T_prefetch > 771319 ns`，超出的部分就是该层 stall
